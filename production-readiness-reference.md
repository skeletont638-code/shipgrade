# What Makes an App Production-Ready — Research-Grounded Reference for shipgrade

> Scope: operational readiness and reliability. Deep application/infrastructure security (authn/authz internals, vuln scanning, threat modeling, crypto) is **out of scope** — owned by a separate security product. Items that border security are included only where they are operational hygiene (e.g., not committing secrets); see §4 for the explicit boundary.

---

## 1. Authoritative frameworks

These are the primary, citable sources the research draws on. Each line states what it uniquely contributes.

- **The Twelve-Factor App** (12factor.net) — the canonical baseline for config-in-environment, dependency declaration, stateless processes, build/release/run separation, dev/prod parity, and logs-as-streams. Defines the portability and disposability properties a deployable app must have.
- **Google SRE Book & SRE Workbook** (sre.google) — the reference for SLIs/SLOs/error budgets, the four golden signals, multi-window burn-rate alerting, the Launch Coordination Checklist, the Production Readiness Review (PRR) engagement model, data-integrity defense-in-depth (Ch. 26), on-call/escalation practice (Ch. 11), DiRT disaster-recovery testing, blameless postmortems, and N+2 capacity.
- **AWS Well-Architected Framework** — Reliability and Operational Excellence pillars: RTO/RPO, Operational Readiness Reviews (ORRs), tested restores, self-healing, small reversible deploys, game days, observability for actionable insight, and the retry anti-patterns (REL05-BP03). Performance Efficiency pillar: connection pooling, caching/edge-caching hit-rate targets, read-replica/read-scaling, indexing, query performance.
- **Azure Well-Architected Framework** — Reliability and Operational Excellence principles: business-anchored RTO/RPO/SLOs, automated self-healing, fault isolation of critical vs. degradable components, immutable transactionally-consistent backups, IaC + safe deployment practices, decoupled monitoring stack.
- **Release It! 2nd Ed. (Michael Nygard, 2018)** — the stability-patterns canon: timeouts, circuit breakers, bulkheads, fail-fast/graceful degradation, and the "Integration Points" failure modes that turn one slow dependency into a cascading outage.
- **AWS Builders' Library** — practitioner deep-dives on timeouts/retries/backoff-with-jitter and idempotent APIs/receivers for safe retries.
- **Production-Ready Microservices (Susan Fowler, O'Reilly 2016)** — Appendix A checklist spanning stability, scalability, fault tolerance, monitoring, and documentation; the most concrete per-service "is it ready" gate.
- **Accelerate / DORA (Forsgren, Humble, Kim, 2018 + State of DevOps reports)** — empirical link between trunk-based development, continuous integration, deployment automation, and delivery performance/stability; source of the "test on every commit," short-lived-branches, config-change-as-leading-CFR-contributor, and AI-needs-testing-fundamentals findings.
- **OpenTelemetry** (docs + semantic conventions 1.42.0) — the three-signals model (traces, metrics, logs), trace/span-ID log correlation, and W3C trace-context propagation.
- **Observability Engineering (Majors, Fong-Jones, Miranda, O'Reilly 2022)** — wide, high-cardinality structured events for exploratory debugging of novel failures.
- **Martin Fowler / Pramod Sadalage — Evolutionary Database Design** — expand-contract (parallel-change) migrations, versioned/sequenced migrations, and testing migrations at production scale.
- **Kubernetes documentation** — liveness/readiness/startup probe semantics, graceful shutdown (SIGTERM, terminationGracePeriod, preStop hooks).
- **OWASP (Secrets Management Cheat Sheet, Input Validation Cheat Sheet, ASVS 4.0)** — operational-hygiene slices used here: no plaintext secrets in code/pipelines, scoped/short-lived pipeline credentials, input validation at trusted enforcement boundaries.
- **Principles of Chaos Engineering** (principlesofchaos.org) — steady-state hypothesis and blast-radius limits for fault-injection experiments.
- **Supporting standards/sources** — RFC 6585 (HTTP 429 + Retry-After); Heroku Dev Center (H12 30 s hard timeout, background jobs/DLQs); web.dev performance budgets + perceived-latency research (the ~500 ms responsiveness budget); database-performance practitioner sources (N+1, connection pooling, cursor pagination, read replicas); **Enterprise Integration Patterns (Hohpe & Woolf)** — Dead-Letter Channel, Idempotent Receiver, Invalid-Message Channel for async messaging; **messaging-platform docs** (Kafka consumer-lag / AWS SQS `ApproximateAgeOfOldestMessage` + redrive/DLQ) for queue saturation and poison-message handling; **Martin Fowler — "Feature Toggles" (Pete Hodgson)** for feature-flag lifecycle/kill-switch hygiene; cache-stampede literature (Vattani et al., probabilistic early expiration); **API-evolution practitioner guidance** (Microsoft REST API Guidelines, Stripe API versioning) for external backward compatibility.

## 2. Grounded checklist by category

Severity defaults are per-item; calibration rules are in §5. Items are verified against the cited source unless marked **(unverified)**.

> **Cross-cutting — SLOs/SLIs/error budgets.** These are scored as one coherent concern even though the checks live in several categories. **See together: 2.2 (define SLIs/SLOs), 2.6 (burn-rate alerting), 6.20 (error-budget deploy freeze), 8.5 (SLO documentation).** §3 item 1 explains the rationale for treating SLO existence/quality as a single first-class dimension.

### Category 1 — Error handling & resilience

| # | Check | Failure mode if absent | Severity | Source |
|---|-------|------------------------|----------|--------|
| 1.1 | **Set explicit timeouts on every remote call** (DNS, TLS handshake, connect, and response). | A slow/unreachable dependency holds threads & connections indefinitely; under load pools fill and the whole service stops serving — classic cascading failure. | **Blocker** | Release It! 2nd Ed. ch. 4 (Timeouts); AWS Builders' Library — Timeouts, retries, and backoff with jitter |
| 1.2 | **Retry only transient errors, with capped exponential backoff + full jitter, at exactly one layer** of the call stack. | No backoff → thundering-herd amplification; multi-layer retries compound (Nygard's 5 tiers × 3 = 243× DB load), preventing recovery. | **Blocker** | AWS Well-Architected REL05-BP03; AWS Builders' Library; Release It! ch. 4; Google SRE Book — Service Best Practices |
| 1.3 | **Cap retries by max count AND max elapsed time.** | Unbounded retry queues grow during an outage and persist after recovery, keeping the system saturated — a metastable failure. | High | AWS Well-Architected REL05-BP03 |
| 1.4 | **Never retry non-idempotent operations** unless idempotency is verified; never retry errors that can't resolve (403, invalid-argument, config errors). | Retried writes cause duplicate side effects; retrying permanent errors wastes capacity and hides the real bug. | High | AWS Well-Architected REL05-BP03 (anti-patterns) |
| 1.5 | **Wrap each integration point in a circuit breaker** (CLOSED/OPEN/HALF-OPEN). | A down dependency consumes threads/connection-pool slots, degrading unrelated paths; breaker short-circuits and gives the dependency time to recover. | High | Release It! ch. 4; AWS Prescriptive Guidance — Circuit breaker pattern |
| 1.6 | **Isolate resource pools per dependency (bulkheads).** | A shared pool lets one slow dependency starve all others; bulkheads bound the blast radius. | High | Release It! ch. 4; Azure Architecture Center — Bulkhead Pattern |
| 1.7 | **Provide fallback / degraded-mode responses** (cached/default/reduced-functionality) when a circuit is open or a non-critical dependency (incl. cache — see 7.14) is unreachable. | Returning raw errors when a fallback was possible converts partial degradation into user-visible failure. | High | Release It! ch. 4 (resilience-engineering lens) |
| 1.8 | **Handle SIGTERM gracefully:** stop accepting new requests, drain in-flight work, exit cleanly. | Abrupt termination during rolling deploys/autoscale-in drops requests mid-response and corrupts long-running jobs. | High | 12factor.net/disposability (Factor IX); Kubernetes Pod Lifecycle docs |
| 1.9 | **Liveness probe checks only truly unrecoverable states** (deadlock/corruption), never external dependencies; failureThreshold ≥ 3. | A liveness probe that checks the DB restarts pods when the DB is merely slow, destroying in-flight requests and cascading restarts across replicas. | High | Kubernetes probe docs; Fairwinds liveness best practices |
| 1.10 | **Degrade gracefully under overload / shed load**; never hard-crash when request rate exceeds capacity. | A service that crashes under overload turns a partial overload into a total outage and cascades to dependents. | High | Google SRE Book — Service Best Practices (Overload) |
| 1.11 | **Modularize so non-critical component failures are isolated** from the critical path (fault isolation). | A monolithic failure domain converts partial degradation into total outage. | High | Azure WAF Reliability Principles — Design for resilience |
| 1.12 | **Automate failure detection + remediation / self-healing** (restart, failover, scale-out) without waiting for a human. | Manual detection + manual fix inflates MTTR from minutes to hours and routinely breaches SLOs before anyone responds. | **Blocker** | AWS WAF Reliability Pillar; Azure WAF Reliability Principles; Fowler, Production-Ready Microservices App. A |
| 1.13 | **Eliminate single points of failure** — redundancy at host, network, and data layers. | A single instance / single-AZ DB / single-region deploy turns any hardware or network event into a full outage. | **Blocker** | Fowler, Production-Ready Microservices App. A |
| 1.14 | **Inventory all dependencies with documented fallbacks/caching for each failure scenario; test those failure paths.** | Undefended dependencies are the most common cascading-failure source; untested failure paths surface first in production. | **Blocker** | Fowler App. A; Google SRE Book — Launch Checklist (Dependencies) |
| 1.15 | **Chaos/fault-injection with a steady-state hypothesis and enforced blast-radius limits**, run in pre-prod and prod. | Resilience mechanisms never exercised against real failures are unverified assumptions; unbounded chaos causes uncontrolled customer impact. | Medium | Principles of Chaos Engineering; AWS WAF Op-Ex; Azure WAF Reliability |
| 1.16 | **Validate config presence and type at startup; fail fast before serving traffic. (unverified)** | Lazy config reads let the app start, pass health checks, then crash hours later on first use of a missing value. | Medium | Contemporary fail-fast startup-validation practice (Pydantic `BaseSettings`, Zod schema parsing, Spring `@ConfigurationProperties` binding validation). **Not endorsed by the 12-Factor manifesto, which does not address startup validation; noted as unverified.** |
| 1.17 | **Handle poison messages on async consumers**: bounded redelivery attempts, then route the message to a dead-letter / invalid-message channel and alert — never let a single un-processable message block the partition/queue. | At-least-once brokers redeliver a message that always fails (bad deserialization, schema mismatch) forever, head-of-line-blocking every subsequent message on the partition until the consumer is manually drained. | High | Enterprise Integration Patterns (Hohpe & Woolf) — Dead-Letter / Invalid-Message Channel; AWS SQS redrive/DLQ docs |

### Category 2 — Observability

| # | Check | Failure mode if absent | Severity | Source |
|---|-------|------------------------|----------|--------|
| 2.1 | **Instrument all three signals: structured logs, metrics, distributed traces.** | Any single signal alone leaves novel-failure debugging to blind guessing — no cross-service flow, no per-request context, no discrete error text. | **Blocker** | OpenTelemetry — Observability Primer; Fowler App. A; Azure WAF Op-Ex (decoupled monitoring stack) |
| 2.2 | **Define quantitative SLIs/SLOs for every user-facing dimension** (availability, latency, throughput) before launch. *(SLO cluster — see also 2.6, 6.20, 8.5.)* | Without a shared definition of "working," outages go undetected and there's no objective basis for deploy/escalation decisions. | **Blocker** | Google SRE Book Ch. 4 (SLOs) |
| 2.3 | **Instrument and alert on every user-visible request failure.** | Silent failures accumulate undetected until they show up as user complaints or churn. | **Blocker** | Google SRE Book Ch. 32 (PRR) |
| 2.4 | **Alert on the four golden signals (latency, traffic, errors, saturation) as primary paging criteria.** | Alerting on causes (CPU%, disk) produces storms on non-events and misses cascading failures that manifest differently from their root cause. | **Blocker** | Google SRE Book — Monitoring Distributed Systems |
| 2.5 | **Alert on symptoms (user-visible impact), not proximate causes.** | Cause-based alerts fire as constant false positives; symptom-based alerts page only when users are actually affected. | High | Google SRE Book — Monitoring Distributed Systems |
| 2.6 | **Use multi-window, multi-burn-rate SLO alerts** (e.g. 14.4×/1h+5m, 6×/6h+30m, 1×/3d+6h). *(SLO cluster — see also 2.2, 6.20, 8.5.)* | Single-threshold alerts either storm or miss slow burns that quietly exhaust the budget. | High | Google SRE Workbook — Alerting on SLOs |
| 2.7 | **Classify every alert as page / ticket / log**; no undifferentiated or email-only alerts; each alert carries severity, owner role, and remediation context. | Undifferentiated, noisy alerts train on-call to ignore pages, so real incidents are missed. | Medium–High | Google SRE Book — Service Best Practices; Azure WAF Op-Ex Principles |
| 2.8 | **Emit structured logs (JSON/key=value) with trace ID + span ID injected into every record.** | Unstructured logs can't be queried at scale or correlated to traces — you can't jump from a log line to the full request trace mid-incident. | High | OpenTelemetry Semantic Conventions 1.42.0; Fowler App. A (correlation IDs) |
| 2.9 | **Propagate trace context (W3C `traceparent`) across all service boundaries** (HTTP, RPC, and async message headers). | Traces break at boundaries into disconnected spans; latency attribution across services becomes impossible. | High | OpenTelemetry — Observability Primer |
| 2.10 | **Apply RED (Rate, Errors, Duration) to every public endpoint.** | Without per-endpoint signals you can't tell which endpoint is degraded → untargeted rollbacks and long MTTR. | High | Google SRE Book/Workbook — Monitoring |
| 2.11 | **Apply USE (Utilization, Saturation, Errors) to every infra resource.** | Saturation is a leading indicator; without it the system is blind to impending failure. | Medium | Google SRE Book — Monitoring Distributed Systems |
| 2.12 | **Use latency histograms (exponential buckets), not averages/single percentiles.** | Averages mask bimodal/tail latency; SLO compliance can't be measured and tail failures go undetected. | Medium | Google SRE Book — Monitoring Distributed Systems |
| 2.13 | **Emit wide, high-cardinality structured events** (user_id, request_id, tenant, flags) for post-hoc exploration. | Pre-aggregated low-cardinality metrics can't answer "which user/tenant/request caused this?" without a new deploy to add instrumentation. | High | Observability Engineering (O'Reilly 2022) |
| 2.14 | **Aggregate/deduplicate errors (e.g. Sentry) with sampling controls.** | Without aggregation, exceptions are lost in volume; without sampling, an error spike overwhelms the error tracker exactly when needed. | High | Sentry Docs — Sampling |
| 2.15 | **Maintain golden-signal dashboards with change-event markers** — annotate not only binary deploys but **IaC apply runs, config/feature-flag pushes, and autoscaling events** (see 6.7) on the same timeline. | Without change markers, the first 10 minutes of every incident is spent ruling out recent changes; an unannotated IaC apply or flag flip is an invisible root cause. | Medium | Google SRE Workbook — Monitoring; AWS/Azure WAF Op-Ex (correlate changes to telemetry) |
| 2.16 | **Add black-box / synthetic external probing** alongside white-box metrics. | Internal metrics can look healthy while DNS, the load balancer, or routing makes the service externally unreachable. | Medium | Google SRE Book — Monitoring Distributed Systems |
| 2.17 | **Validate alert rules + routing in non-prod before deploy** (metrics → rule eval → routing). | A broken routing rule discovered during an incident means nobody is paged. | Medium | Google SRE Workbook — Monitoring |
| 2.18 | **Expose a dedicated `/ready` endpoint** reflecting real traffic-serving ability (warm-up + drain), separate from liveness. | Conflating readiness with liveness gives the orchestrator no signal to stop routing to an initializing/draining pod. | Medium | Kubernetes probe docs |
| 2.19 | **Write logs to stdout/stderr as an unbuffered stream**; the app does not manage files/rotation/routing. | File-logging loses history when the container/VM is replaced; buffering loses the last seconds before a crash. | High | 12factor.net/logs (Factor XI) |
| 2.20 | **Monitor TLS-certificate expiry and alert with ≥30-day lead time** (and renew automatically where possible). | Expired certificates are a common, fully avoidable hard outage: every client connection fails TLS at once while internal app metrics still look green. | High | Google SRE Book — Monitoring (black-box); practitioner cert-expiry monitoring (cert-manager / ACME automation) |
| 2.21 | **Treat queue depth and consumer lag as a first-class saturation signal** for every async consumer; alert on lag/oldest-message-age thresholds. | A consumer that is technically alive but falling behind never trips a USE/CPU alert; backlog grows silently until SLA-breaching processing delay or broker retention drops un-consumed messages. | High | Messaging-platform monitoring (Kafka consumer-lag, AWS SQS `ApproximateAgeOfOldestMessage`); Google SRE — Monitoring (saturation) |
| 2.22 | **Retain logs long enough to cover the investigation window (typically ≥30 days for prod) and ensure they are centrally indexed and queryable in seconds.** | Streaming logs (2.19) that expire in 7 days or can't be searched are useless when an incident is diagnosed 8 days later — a direct MTTR multiplier. | Medium | Google SRE Book — practices of logging/centralized log search; practitioner log-retention guidance |
| 2.23 | **Expose a structured health endpoint** returning build/version, startup time, and per-dependency status (not just HTTP 200). | A bare liveness 200 can't answer "is this the right build?" or "which dependency is down?" during an incident without log access, slowing triage. | Medium | Kubernetes probe docs; practitioner health-endpoint patterns (Spring Boot Actuator, MicroProfile Health) |

### Category 3 — Config & secrets hygiene

| # | Check | Failure mode if absent | Severity | Source |
|---|-------|------------------------|----------|--------|
| 3.1 | **All config (credentials, hostnames, per-deploy values) in environment variables — never hardcoded or committed.** | Committed credentials leak on open-sourcing or machine compromise; hardcoded endpoints force code changes to deploy to new environments. | **Blocker** | 12factor.net/config (Factor III) |
| 3.2 | **Secrets never in source, Dockerfiles, or image layers; injected at runtime from a secrets manager** (Vault, AWS/GCP Secrets Manager). | Baked-in secrets are visible to anyone with repo/registry read access and persist in git/image history forever — the leading CI/CD credential-compromise cause. | **Blocker** | OWASP Secrets Management Cheat Sheet |
| 3.3 | **Pipeline credentials are least-privilege and auto-expire after the job; no long-lived static tokens.** | An over-permissioned, permanent CI token on a compromised runner can exfiltrate all secrets and deploy malicious artifacts everywhere. | High | OWASP Secrets Management Cheat Sheet |
| 3.4 | **Backing services (DB, cache, queue, external APIs) referenced only via config — swappable without code changes.** | Coupling a specific host/impl into code means a failover also requires an emergency code deploy. | High | 12factor.net/backing-services (Factor IV) |
| 3.5 | **Validate config inputs (syntax + semantics) before applying; on bad input retain the previous valid state and alert — never crash or reset to defaults.** | A bad config push that replaces good state is a top cause of self-inflicted outages. | High | Google SRE Book — Service Best Practices (Fail Sanely) |
| 3.6 | **Manage each config variable independently — do NOT group config into named env batches** ("staging"/"production" groups). | Env-group schemes scale badly: each new deploy context needs a new group and combinatorial mappings → drift and misconfiguration. | Medium | 12factor.net/config (Factor III) |
| 3.7 | **Cover configuration changes with automated tests of equivalent gate weight to code changes.** | Config changes are a leading cause of distributed-system outages; untested config is a class of change with zero quality gate. | High | **DORA / Accelerate (config changes as a leading change-failure-rate contributor) and AWS WAF Op-Ex (treat config changes with the same rigor as code) as primary support; Cindy Sridharan, "Testing Microservices: A Sane Approach" (2017) as a corroborating practitioner source, not a primary framework.** |

### Category 4 — Reliability & data integrity

| # | Check | Failure mode if absent | Severity | Source |
|---|-------|------------------------|----------|--------|
| 4.1 | **Define and document explicit RTO and RPO per critical flow, negotiated with business stakeholders.** | Without recovery targets there's no acceptance criterion for DR design; recovery is unguided and SLAs are violated. | **Blocker** | AWS WAF REL13-BP01; Azure WAF Reliability Principles |
| 4.2 | **Tiered backups distinct from replication** (frequent local + mid-tier distributed + offsite archive); immutable and transactionally consistent. | Replication faithfully propagates corruption/deletion — it is not recoverability. | **Blocker** | Google SRE Book Ch. 26; Azure WAF Reliability Principles |
| 4.3 | **Automate and regularly exercise full end-to-end restores; verify actual RTO/RPO are met** (not just "backup exists"). | An untested backup isn't a backup; corruption, permission, or schema-drift failures only surface during a real disaster. | **Blocker** | AWS WAF REL13-BP03; Google SRE Book Ch. 26 (DiRT) |
| 4.4 | **Make non-idempotent mutating operations idempotent via caller-supplied tokens stored atomically with the result.** | Without idempotency, a retried payment/order/mutation produces duplicate effects (double charge, duplicate record). | **Blocker** | AWS Builders' Library — Idempotent APIs; AWS WAF REL04-BP04 |
| 4.5 | **Stateless, share-nothing processes** — no session/upload/derived state in local memory or disk across requests. | In-process state is destroyed on every deploy/crash/scale event; sticky sessions mask this until rolling deploys, and block autoscaling. | High | 12factor.net/processes (Factor VI); AWS WAF Performance Efficiency |
| 4.6 | **Apply schema migrations with expand-contract (parallel change) for backward compatibility across rolling deploys.** | Old and new code run together mid-deploy; a breaking change makes old code fail immediately → corruption or downtime. | High | Fowler/Sadalage — Evolutionary Database Design; Google CRE — Reliable Releases & Rollbacks |
| 4.7 | **Version-control migrations with sequential IDs + an applied-migrations changelog table.** | Untracked migrations make staging/prod/dev schemas silently diverge → invalid tests, failed/corrupting deploys. | High | Fowler/Sadalage — Evolutionary Database Design |
| 4.8 | **Test migrations against production-scale data volumes on staging before prod.** | An ALTER that's instant on 1k dev rows can lock a 100M-row prod table for minutes. | High | Fowler — Evolutionary Database Design; PlanetScale safe-schema-change docs |
| 4.9 | **Wrap multi-step state mutations in transactions (atomicity); never leave partial-write state.** | Crashes/network drops mid-operation leave rows partially updated → silent corruption read downstream. | High | ACID atomicity/consistency principle (database literature) |
| 4.10 | **Soft-delete with a 30–90 day retention window before permanent destruction.** | Application bugs cause most data-loss; hard deletes propagate irreversibly to all replicas with no first line of defense. | High | Google SRE Book Ch. 26 |
| 4.11 | **Run continuous out-of-band data-validation pipelines (invariants, referential integrity, counts); alert within 24h of corruption.** | "Bad data doesn't sit idly by — it propagates"; detection latency directly bounds blast radius. | High | Google SRE Book Ch. 26 |
| 4.12 | **Validate all input at every trust boundary (API entry, inter-service, queue consumers) on a trusted enforcement tier.** | Unvalidated input crossing a boundary is the root cause of most corruption bugs and constraint violations; client-side-only validation is trivially bypassed. | High | OWASP ASVS 4.0; OWASP Input Validation Cheat Sheet |
| 4.13 | **Structured, tested, documented recovery plans for every stateful component, exercised via regular drills.** | Untested recovery plans fail under real-incident pressure (gaps in procedure, permissions, tooling). | High | Azure WAF Reliability Principles — Design for recovery |
| 4.14 | **Provision N+2 capacity** — meet SLO with the two largest instances unavailable (one planned + one unplanned). | N+1 fails when an unplanned failure coincides with maintenance, dropping below SLO. | High | Google SRE Book — Launch Checklist / Service Best Practices |
| 4.15 | **Document and enforce data-retention policies; automate purge jobs and verify completion within compliance windows.** | Paper-only retention creates the same liability as none; silently failing purge jobs are equally dangerous. | Medium | Google SRE Book Ch. 26 |
| 4.16 | **Manage/detect configuration drift at the DR site; keep DR deployable and current.** | A drifted DR site (stale images, outdated secrets) won't accept failover even if networking works. | Medium | AWS WAF REL13-BP04 |
| 4.17 | **Automate recovery/failover workflows so failover needs no manual steps under time pressure.** | Manual recovery fails under stress; coordination overhead alone can blow the RTO. | Medium | AWS WAF REL13-BP05 |
| 4.18 | **Write async consumers to be idempotent (Idempotent Receiver) whenever the broker guarantees at-least-once delivery** — dedupe on a message/business key stored atomically with the effect. | At-least-once brokers will redeliver on rebalance, retry, or ack-timeout; a non-idempotent consumer double-processes (duplicate charges, double-applied events). 4.4 covers the synchronous API edge; this covers the queue-consumer edge. | High | Enterprise Integration Patterns (Hohpe & Woolf) — Idempotent Receiver; AWS Builders' Library — Idempotency |

### Category 5 — Testing

| # | Check | Failure mode if absent | Severity | Source |
|---|-------|------------------------|----------|--------|
| 5.1 | **Run automated unit + acceptance tests on every commit to version control.** | Regressions accumulate silently between commits; large untested batches are the primary driver of high change-failure rate. | **Blocker** | Accelerate ch. 4; DORA Continuous Integration capability |
| 5.2 | **Follow the test pyramid (many unit → fewer integration → minimal E2E), not an hourglass.** | Heavy/thin-integration E2E pipelines are slow and flaky; eroded trust makes developers bypass CI, removing the gate. | High | Google Testing Blog — Fixing a Test Hourglass (2020) |
| 5.3 | **Services independently testable without a fully integrated environment.** | If every test needs the whole dependency graph, cycles slow to hours and block fast feedback — a predictor of low deploy frequency. | High | Accelerate ch. 5 |
| 5.4 | **Consumer-driven contract tests gate every API boundary** between services. | Without them, a broken contract is found only in E2E or production, as a cascading failure with many suspect services. | High | Martin Fowler — Testing Strategies in a Microservice Architecture |
| 5.5 | **Integration tests exercise error/timeout/degraded paths, not just happy paths.** | Untested failure paths are discovered in production under real load, often as cascading outages. | High | Martin Fowler — Testing Strategies in a Microservice Architecture |
| 5.6 | **Bound the E2E suite to a minutes-scale time budget, scoped to critical user journeys; ensure data independence via automated setup.** | Unbounded E2E delays feedback past developer context; data coupling causes false failures that mask real regressions. | Medium | Martin Fowler — Testing Strategies in a Microservice Architecture |
| 5.7 | **Lint + unit + integration + E2E coverage, plus load and chaos testing for resiliency.** | Unit tests don't surface integration breakage or degraded-under-load behavior; untested failure paths run first in incidents. | High | Fowler, Production-Ready Microservices App. A |
| 5.8 | **Maintain robust automated testing as AI-assisted code generation scales.** | AI raises output velocity and the rate of untested/subtly-wrong code; relaxing testing while adopting AI degrades stability metrics. | Medium | DORA State of DevOps Report 2024 |
| 5.9 | **Test both flag-on and flag-off code paths in CI for every active feature flag** (see 6.21); a canary (6.5) is not a substitute. | A flag introduces a combinatorial state space; untested combinations ship dead-on-arrival behavior when the flag is flipped — the canary only exercises whatever state is currently configured. | Medium | Martin Fowler — "Feature Toggles" (Pete Hodgson) |
| — | *Flaky-test quarantine and smoke/canary gates are operational deploy gates; see Category 6 (6.10, 6.5, 6.11).* | | | |

### Category 6 — CI/CD & deploy

| # | Check | Failure mode if absent | Severity | Source |
|---|-------|------------------------|----------|--------|
| 6.1 | **Every commit to trunk triggers an automated build + full test suite completing within ~10 minutes.** | Without a fast gate, broken code accumulates until late testing or production; deploy frequency collapses. | **Blocker** | DORA CI capability; Accelerate ch. 4 |
| 6.2 | **Fully automated pipeline from commit to production; no manual deploy steps.** | Manual steps introduce human error, extend recovery, and break deploy-on-demand. | **Blocker** | DORA Continuous Delivery capability; DORA 2024 Report; Fowler App. A |
| 6.3 | **Strict build/release/run separation: immutable, uniquely-identified, retrievable release artifacts (append-only ledger).** | Mutating code at runtime makes deploys unreproducible and un-rollbackable — a 3am outage can't be reverted. | **Blocker** | 12factor.net/build-release-run (Factor V) |
| 6.4 | **Short-lived branches (merged < 1 day); fewer than ~3 active branches.** | Long-lived branches create large risky merges, delay defect discovery, and block continuous deployability. | High | DORA Trunk-Based Development; Accelerate ch. 4 |
| 6.5 | **Progressive/canary rollout: 1–5% of traffic first, with automated metric comparison vs. control and automated rollback triggers.** | A full-fleet deploy of a bad release hits 100% of users before detection; canary bounds blast radius and enables auto-rollback. | High | Google SRE Workbook — Canarying Releases; Google SRE/AWS/Azure Op-Ex |
| 6.6 | **Documented, tested rollback plan for every deploy; rollback is the default first response to any anomaly.** | Investigate-before-rollback extends outages; untested rollback fails under pressure (esp. with schema changes). | High | Google CRE — Reliable Releases and Rollbacks |
| 6.7 | **All infrastructure as code, version-controlled, deployed via the same pipeline; no manual prod changes — and emit IaC-apply / autoscale / config-push events into the observability layer (see 2.15).** | Manual changes cause config drift (DORA: 2.3× higher change-failure rate) and make DR recreations inconsistent; un-annotated infra changes are invisible incident root causes. | High | DORA IaC capability; Azure WAF Op-Ex; AWS WAF Op-Ex |
| 6.8 | **Deploy small, frequent, reversible increments — never large batched releases without rollback.** | Large batches widen blast radius, obscure root cause, and lengthen recovery. | High | AWS WAF Op-Ex; Azure WAF Op-Ex |
| 6.9 | **Hermetic, reproducible builds: same SHA → bit-for-bit identical artifacts on different machines.** | Non-reproducible builds prevent verifying that the tested artifact is what shipped, and prevent reliable rollback to a known-good binary. | High | Google SRE Book — Release Engineering |
| 6.10 | **Quarantine or delete flaky tests before they reach the CI critical path.** | Flaky reds train developers to ignore CI; the quality gate becomes ineffective. (Google found that **of test failures associated with flaky tests, ~84% were attributable to flakiness rather than a real code defect** — flaky tests are the dominant source of false CI failures at scale.) | High | Google Testing Blog — Flaky Tests at Google (2016) |
| 6.11 | **Run smoke tests automatically against every production deploy; gate rollout continuation on them.** | Env-specific config/TLS/DNS/infra differences let a deploy that passed all pre-prod gates serve broken traffic until a human notices. | **Blocker** | Google SRE Book — Testing for Reliability |
| 6.12 | **Declare a HEALTHCHECK (Docker) or liveness + readiness probes (Kubernetes) for every containerized service.** | Without probes, a started-but-not-ready or deadlocked container still receives traffic; the orchestrator can't restart or hold. | High | Kubernetes — Configure Probes docs |
| 6.13 | **Add a preStop hook with a short sleep (5–10s) to bridge endpoint-removal propagation vs. SIGTERM.** | LB propagation lags 1–3s; without the sleep the container rejects connections before routing catches up → 502/connection-refused bursts every deploy. | Medium | Kubernetes graceful-shutdown / preStop guides |
| 6.14 | **Container images run as a non-root user (explicit USER, UID ≠ 0).** | A container escape/RCE as root immediately yields host root; non-root bounds the blast radius. | High | Sysdig — Dockerfile best practices |
| 6.15 | **Pin container base images by SHA digest, not mutable tag.** | Mutable tags silently change content between builds, introducing unreviewed changes/vulns with no code change. | Medium | Docker image security best practices |
| 6.16 | **Complete dependency declaration in a lockfile + runtime isolation (no implicit system packages).** | Undeclared deps cause "works on my machine" crashes when a host library is absent in the prod container. | High | 12factor.net/dependencies (Factor II) |
| 6.17 | **Dev and prod use the same type and version of every backing service** (no adapter-abstracted substitutes). | SQLite-local vs. Postgres-prod causes dialect, constraint, and isolation bugs that pass local tests and corrupt prod. | High | 12factor.net/dev-prod-parity (Factor X) |
| 6.18 | **One codebase per app in version control; share logic via versioned libraries, not copy-paste.** | Multiple codebases force cross-repo coordination for one fix; copy-pasted code drifts into split behavior. | Medium | 12factor.net/codebase (Factor I) |
| 6.19 | **Schema migrations decoupled from binary deploys (expand-contract) so code can roll back without schema incompatibility.** | Atomic schema+binary deploys make rollback impossible — the old binary can't run against the new schema. | High | Google CRE — Reliable Releases and Rollbacks |
| 6.20 | **Enforce a deploy freeze (non-P0/security only) when the error budget for the rolling window is exhausted.** *(SLO cluster — see also 2.2, 2.6, 8.5.)* | Shipping features into an already-degraded service compounds degradation and breaks the velocity/reliability accountability link. | High | Google SRE Book/Workbook — Error Budget Policy |
| 6.21 | **Operate feature flags with lifecycle hygiene: maintain a runtime kill switch for risky features, and track + retire stale flags on a defined schedule.** | A kill switch is the fast-remediation complement to rollback (6.6) — disable a bad feature in seconds without redeploying; meanwhile abandoned flags accumulate dead/untested code paths and combinatorial behavior nobody validates. | High | Martin Fowler — "Feature Toggles" (Pete Hodgson); Google SRE — progressive rollout / kill-switch practice |
| 6.22 | **Manage external API breaking changes explicitly: additive-only backward-compatible evolution by default, versioned paths/headers for breaking changes, published deprecation windows, and consumer notification.** | For any API with external/third-party consumers, an undocumented breaking change is a reliability incident *for them*; 5.4 covers internal contracts but not the public surface. | High | Microsoft REST API Guidelines; Stripe API-versioning practice; Fowler — evolutionary API design |

### Category 7 — Performance & scale

| # | Check | Failure mode if absent | Severity | Source |
|---|-------|------------------------|----------|--------|
| 7.1 | **Use a database connection pool; never open a raw connection per request.** | Per-request connections cost 10–50ms and exhaust the DB's hard connection limit under load → cascading connection-refused errors. | High | Stack Overflow Blog — Connection pooling; AWS WAF PERF03-BP02 |
| 7.2 | **Detect and eliminate N+1 query patterns before production.** | N+1 issues one round-trip per row — invisible on 5 dev rows, catastrophic on 5,000 prod rows (100ms → multi-second timeout). | High | PlanetScale — N+1 Query Problem; ScoutAPM |
| 7.3 | **Index every foreign-key column and every column used in WHERE/ORDER BY/GROUP BY on high-traffic queries.** | An unindexed filter triggers a full sequential scan per request, saturating I/O and collapsing throughput at scale. | High | AWS WAF PERF03-BP04 (anti-patterns) |
| 7.4 | **Use cursor/keyset pagination for large or frequently-updated result sets.** | Offset pagination counts and discards all preceding rows per page → full-table scan at deep pages; also skips/duplicates rows under concurrent inserts. | High | Gusto Embedded — API Pagination; Stacksync — Keyset vs Offset |
| 7.5 | **Offload work routinely exceeding the ~500 ms responsiveness budget to an async background queue with a dead-letter queue.** | Synchronous long work blocks web threads (and approaches Heroku's H12 30 s hard timeout) and stops accepting connections; without a DLQ, exhausted-retry jobs are silently lost. | High | web.dev performance budgets + perceived-latency research (~500 ms responsiveness target); Heroku Dev Center — Background Jobs and Queueing (H12 30 s timeout, DLQs) |
| 7.6 | **Enforce API rate limiting; return HTTP 429 + Retry-After when limits are exceeded.** | One misbehaving/compromised client can monopolize capacity; standardized 429+Retry-After lets clients back off instead of retry-storming. | High | RFC 6585; Speakeasy — Rate Limiting Best Practices |
| 7.7 | **Implement a remote cache (e.g. Redis) for read-heavy services; target ≥80% hit rate, with monitoring.** | Without caching every read hits the primary DB; a hit rate <80% signals an undersized cache or non-cacheable pattern needing redesign. | Medium | AWS WAF PERF03-BP05 |
| 7.8 | **Define a cache-invalidation strategy (TTL or event-driven) for every cached dataset.** | No TTL/invalidation serves stale data indefinitely; AWS marks "treating cache as durable/always-available" an explicit anti-pattern. | Medium | AWS WAF PERF03-BP05 (anti-patterns) |
| 7.9 | **Stateless application tier so horizontal autoscaling adds/replaces instances without data loss or session-routing complexity.** | Local in-memory sessions silently log users out or route to wrong state when an instance is added/replaced; externalized state is a prerequisite for autoscaling. | High | AWS WAF Performance Efficiency Pillar |
| 7.10 | **Run load tests at realistic production scale before every significant launch or infra change; document capacity thresholds and a 6-month growth projection with growth-triggered alerts.** | Theoretical estimates underestimate real load; services that pass at low traffic fail at the first ramp, and growth-triggered outages are predictable. | High | Google SRE Book — Service Best Practices / Launch Checklist |
| 7.11 | **Enable the slow-query log and instrument queries-per-request; alert on queries >100ms and on per-request query-count spikes.** | Slow queries and N+1 regressions are invisible without instrumentation and surface as cascading DB saturation only after load increases. | Medium | AWS WAF PERF03-BP04 / PERF03-BP03 |

## 3. Recommended additions to shipgrade's taxonomy

These surfaced repeatedly across lenses and don't sit cleanly in one of the 8 categories. Recommendation for each:

1. **SLOs / SLIs / error budgets as a first-class concern** — *Recommend: elevate to an explicit cross-cutting dimension. **Implemented in this document** via the SLO-cluster callout at the top of §2 and inline `(SLO cluster — see also …)` cross-references on 2.2, 2.6, 6.20, and 8.5.* Nearly every framework (Google SRE, Fowler, Azure/AWS WAF) treats quantitative reliability targets as the prerequisite that makes "production-ready" measurable. shipgrade should score SLO existence/quality as one coherent thing rather than as four disconnected line items. **Strong recommend.**

2. **Backups & disaster recovery (RTO/RPO, tested restores, DR drills)** — *Recommend: keep folded into "Reliability & data integrity," but make it a clearly-labeled sub-section with its own Blocker checks (4.1–4.4, 4.13) plus the human game-day drill (8.10).* These are among the most consequential Blockers in the corpus ("replication is not recoverability"; "an untested backup is not a backup") and risk being under-weighted if buried among general data-integrity items. Folding is fine; visibility is the fix.

3. **Capacity planning & N+2 provisioning** — *Recommend: fold across "Performance & scale" (load testing, growth projections, CDN/read-replica scaling) and "Reliability & data integrity" (N+2 redundancy), with explicit cross-reference.* It genuinely spans both — runtime throughput vs. survivability with instances down. No new category needed, but the audit should check both halves.

4. **Graceful shutdown + readiness/liveness probe correctness** — *Recommend: keep distributed (Error handling 1.8–1.9, CI/CD 6.12–6.13, Observability 2.18, 2.23) but treat as a named "deploy-time correctness" checklist that must all pass together.* Multiple lenses independently flagged that getting any one of probes/SIGTERM/preStop/readiness wrong drops in-flight requests on every rolling deploy. It deserves emphasis as a cluster even if it stays folded.

5. **Operational Readiness Review (ORR) / PRR as a process gate** — *Recommend: add as a checklist item under Operational docs (8.4), not a new category.* It's a ceremony, not a technical surface; a single "ORR performed and signed off" check captures it.

6. **Supply-chain / artifact integrity (image CVE scanning gate, SBOM, reproducible-build verification)** — *Recommend: mostly defer to the separate security product; keep only the operational-hygiene slices already in CI/CD (non-root, digest-pinned, reproducible builds, no baked secrets).* Drawing the line here keeps the security boundary clean (see §4).

7. **Fallback / degraded-mode behavior** — *Recommend: fold into "Error handling & resilience" (1.7) with the cache-down facet in Performance (7.14).* It's a distinct concern from generic error handling (return cached/default vs. raw error) but belongs to the same resilience family.

8. **Retry/resilience-pattern observability (alerting on retry bursts, open circuits, consumer lag)** — *Recommend: fold into Observability as a specific signal class — consumer lag/queue depth is now an explicit named signal (2.21).* Resilience signals need a home in monitoring; no new category warranted.

9. **Async / event-driven messaging as a cross-cutting concern** — *Recommend: fold the new checks across categories rather than create a "messaging" category — poison-message handling (1.17), consumer-lag/queue-depth signal (2.21), idempotent at-least-once consumers (4.18).* Any service on Kafka/SQS/RabbitMQ has saturation, redelivery, and duplicate-processing failure modes with no analog in the HTTP-request model; the audit should branch these checks on "service consumes from a broker."

10. **Feature-flag operational hygiene** — *Recommend: fold into CI/CD (kill switch + stale-flag lifecycle, 6.21) with a testing note in Cat 5 (5.9).* Flags are now standard deploy infrastructure and the kill switch is the fast-remediation complement to rollback; no new category needed.

11. **External API versioning / backward-compatibility** — *Recommend: add to CI/CD as change-management (6.22).* Distinct from internal contract tests (5.4) and DB expand-contract (4.6/6.19): it governs the **public** surface, where an undocumented breaking change is a reliability incident for third-party consumers.

12. **Continuous profiling, data lineage/audit trails, CDC for point-in-time recovery, telemetry freshness SLA, performance-budget CI gates, mutation/property-based testing** — *Recommend: leave OUT of the core audit for now (note as "advanced/maturity" items).* Each is real but is an advanced-maturity practice that few pre-production teams will have; including them risks drowning Blockers in noise. Revisit if shipgrade adds a maturity-tier scoring model.

**Items explicitly noted by the research as NOT fitting the 8 technical categories (organizational/cultural — recommend leaving out of a technical audit):** commit-to-deploy time-gap discipline (12-Factor), port-binding self-sufficiency (Factor VII), process-type concurrency model (Factor VIII), the SRE ≤50% ops-work cap, error-budget policy as a binding governance document, shared on-call responsibility *model* (note: on-call *rotation correctness*, 8.9, IS in scope as a technical/operational check), architecture-review cadence, and service-catalog/dependency-graph visibility (Backstage-style). These are team-structure/governance concerns outside an app-level technical audit.

## 4. Deliberately out of scope (owned by the separate security product)

The boundary: shipgrade audits **operational hygiene**, not **adversarial security**. Where a control is purely about resisting an attacker, it is out of scope. Explicitly excluded:

- **Authentication & authorization internals** — session management correctness, token/JWT validation, OAuth flows, privilege-escalation/IDOR analysis.
- **Vulnerability scanning & CVE triage as a security gate** — deep dependency vuln analysis, container image CVE severity assessment, SAST/DAST. (shipgrade keeps only the *operational* slices: digest-pinned images, non-root user, no committed secrets.)
- **Threat modeling, attack-surface analysis, and blast-radius-of-compromise** (as distinct from blast-radius-of-failure).
- **Cryptography** — encryption at rest/in transit, key management, certificate provisioning correctness. (Certificate *expiry monitoring/alerting* is in scope as an availability/observability check, 2.20; certificate *issuance/trust/crypto correctness* is the security product's.)
- **Input validation as injection defense** — note: shipgrade includes input validation at trust boundaries (4.12) for **data-integrity/corruption** reasons; the *injection-attack* dimension (SQLi/XSS/command injection) is the security product's.
- **Secrets management as access control** — shipgrade checks that secrets aren't committed and that pipeline creds are short-lived/scoped (operational hygiene, 3.1–3.3); secret *rotation policy depth*, KMS/HSM design, and least-privilege IAM modeling belong to security.
- **Network security** — WAF rules, firewall/security-group policy, DDoS protection design, mTLS configuration. (Rate limiting, 7.6, is in scope as an *availability/overload* control, not as DDoS defense.)
- **Compliance & audit posture** — SOC2/ISO/PCI control mapping, data-residency/regulatory controls beyond operational retention enforcement (4.15).

Where an item appears in both worlds (e.g., input validation, rate limiting, secrets, cert expiry), shipgrade scores the **operational/reliability** facet and defers the **adversarial** facet to the security product.

## 5. Severity calibration notes

Severity reflects **blast radius × likelihood × reversibility** for production readiness, grounded in how the sources frame each control. Use these rules:

**Blocker** — Ship-stopping. The absence makes a class of production failure *inevitable, undetectable, or unrecoverable*, or removes the ability to respond to an incident at all. Calibration signals:
- The failure is **unrecoverable** (no tested restore, no immutable backup → permanent data loss; no versioned release artifact → can't roll back).
- The failure is **undetectable** (no SLOs/golden-signal alerting → outages surface only as user complaints).
- The control is a **prerequisite for every other safeguard** (no automated pipeline / no monitoring before deploy / config & secrets committed to source).
- Sources that explicitly frame the item as foundational: Google SRE "monitoring is the most fundamental requirement"; SRE Book Ch. 26 "replication is not recoverability"; 12-Factor "impossible to make changes to code at runtime."
Examples: 1.1, 1.12–1.14, 2.1–2.4, 3.1–3.2, 4.1–4.4, 5.1, 6.1–6.3, 6.11.

**High** — Serious, likely to cause a real incident, but the system can still be operated and recovered. The failure is *probable under realistic load or during routine deploys*, materially raises blast radius or MTTR, but a competent on-call can mitigate. Most resilience patterns (circuit breakers, bulkheads, backoff caps, poison-message handling, idempotent consumers), canary/rollback discipline, IaC, expand-contract migrations, kill switches, external-API versioning, N+1/index/connection-pool/cache-stampede fixes, cert-expiry and consumer-lag alerting, and per-service runbooks/on-call correctness sit here. This is the default for "best practice that prevents a foreseeable outage."

**Medium** — Degrades reliability, increases toil, or extends MTTR, but rarely causes an outage on its own; often a *contributing factor or detection/efficiency gap*. Examples: cache hit-rate/invalidation tuning, CDN/read-replica offload, USE-method saturation metrics, histogram latency, preStop sleep, digest-pinned tags, config-group hygiene, log-retention/queryability, structured health-endpoint content, flag-state testing, SLO documentation completeness, game-day cadence, blameless-postmortem process. Frequently "would have caught it sooner / made the fix safer" rather than "would have prevented it."

**Low** — Polish, maturity, or future-proofing. Best-practice refinements whose absence is tolerable in early production and which mostly matter at scale or organizational maturity. (Most §3 "leave out / advanced" items — continuous profiling, mutation testing, data lineage — would land here if scored at all.)

**Cross-cutting calibration guidance:**
- **Reversibility dominates.** Anything that makes a bad change *irreversible* (atomic schema+binary deploys, hard deletes, mutable releases, non-idempotent consumers under at-least-once delivery) ranks at least one level higher than the same concern when reversible.
- **Detectability is a Blocker multiplier.** A failure mode you cannot observe (silent data corruption, unmonitored user-facing errors, a lagging-but-alive consumer, a soon-to-expire cert) is treated as more severe than an equally-likely failure you would page on.
- **Context can raise, rarely lower.** A Medium item becomes High for a stateful, high-traffic, or multi-tenant service (e.g., connection pooling, cache invalidation, read-replica routing). Down-rate only when an item is demonstrably inapplicable to the architecture (e.g., probes for a non-orchestrated single-process deploy; queue checks 1.17/2.21/4.18 for a service with no async consumers; 6.22 for an API with no external consumers).
- **Single points of failure and "untested X" are near-automatic escalations.** The corpus repeatedly emphasizes that *untested* backups, recovery plans, rollbacks, runbooks, and on-call routing fail exactly when needed (AWS REL13-BP03: "the only error recovery that works is the path you test frequently"); treat "exists but never exercised" close to "absent."
